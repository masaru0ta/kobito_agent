# kobito_agent

マルチAIエージェント基盤。自律的に思考し成長するAIエージェントを動かすためのシステム。

## できること

1. **記憶を持つAIエージェントを、ディレクトリを作るだけで何体でも動かせる**
2. **ブラウザから会話・設定・記憶・成果物すべてを操作できる**
3. **エージェント同士が自律的に思考し、会話し、Markdownで成果物を残す**

## 設計思想

- エージェントの実体 = プロンプト（意思決定基準）+ ディレクトリ上のデータ（記憶・成果物）
- LLMは交換可能な思考エンジン。UIは交換可能な接点。代替できないのは記憶だけ
- 人格とは、意思決定基準と記憶
- 細かいルールで縛らず、仕組みをシンプルに保つ。エージェント自身の判断に委ねる
- フォールバック処理の禁止。エラーは代替処理で誤魔化さず、明確に報告する

## アーキテクチャ

### ディレクトリ構成
```
kobito_agent/
  team.md                # エージェント一覧と共通ルール
  agents/
    {agent_name}/
      CLAUDE.md            # 各エージェントのシステムプロンプト
      config.yaml          # モデル、名前、トリガー設定
      mission.md           # 目的、方針、継続的な責務
      task.md              # 今やるべき具体的な作業
      memory/              # Mem0データ（エージェント専用）
      output/              # 成果物・ナレッジ(.md)
        index.md           # 成果物の目次
      chat_history/        # 会話履歴
      log/                 # 思考・実行ログ
  server/
    app.py               # Webサーバー（エントリーポイント）
    static/              # HTML/CSS/JS
    api/                 # REST API
  docs/                  # 仕様書
```

## エージェント

- ディレクトリを作るだけでエージェントを追加できる
- エージェント数に制限なし
- 各エージェントは完全に独立した思考コンテキストを持つ（人格の分離）
- 1つのLLM呼び出しで2役を演じさせない
- 相手の内面は見えない。見えるのは相手が発した言葉だけ
- エージェント間で記憶は共有しない。知りたければトリガーで相手に聞く
- `team.md` にエージェント一覧と共通ルールを記載。エージェント追加時はここに1行足す

## 記憶（Mem0）

- 各エージェント専用のストレージ。`agents/{name}/memory/` に格納
- 定期的に検索と記録が行われる
- 何をどう記憶するか細かく規定しない。エージェント自身の判断に委ねる
- 思考サイクルの冒頭で関連記憶をベクトル検索で自動想起

## エージェントの自己管理ファイル

思考サイクルの冒頭で必ず読む。エージェント自身が必要に応じて更新する。

- **mission.md** — 目的、方針、継続的な責務。めったに変わらない
- **task.md** — 今やるべき具体的な作業。missionから導出される。頻繁に変わる

## 成果物

- .mdファイルで `agents/{name}/output/` に出力
- **index.md** を必ず持つ。全ファイルへのリンクと一行説明
- ファイルを作ったらindex.mdを更新する
- 既存のファイルがあれば新規作成より更新を優先する
- 1トピック1ファイル
- ブラウザ + Markdown Reader拡張（Edge）で閲覧

## トリガー

| 種類 | 発火条件 | 例 |
|------|---------|-----|
| ユーザーチャット | Web UIからメッセージ送信 | 「これ調べて」 |
| 定期トリガー | cron式のスケジュール | 10分ごとに自律思考 |
| エージェント間 | 他エージェントからの呼び出し | アダムがエデンに質問 |

## 動作フロー

### 自律思考（定期トリガー）
1. トリガー発火 → runner起動
2. mission.md、task.md を読む
3. 受信メッセージを確認
4. Mem0から関連記憶を自動想起
5. プロンプトを組み立ててLLMを呼ぶ
6. 出力を解析 → 思考ログ保存・メッセージ配送・成果物更新
7. 必要に応じてmission.md、task.mdを更新
8. 会話・思考から記憶を保存

### チャット（ユーザー × エージェント）
1. ユーザーがWeb UIで入力
2. Mem0から関連記憶を自動想起
3. LLMに渡す（想起した記憶 + 会話履歴）→ 応答
4. 応答から記憶を保存
5. Web UIに返す

### コード修正（Web UI → Claude Code）
1. ユーザーがWeb UIで修正指示
2. サーバがClaude Codeをヘッドレス起動（`claude -p`）
3. Claude Codeがファイル操作・シェル実行
4. 結果をWeb UIに返す

## Web UI

ユーザーはここで全て完結する。

- **チャット**: エージェントとの会話。1対1、複数人
- **設定**: config.yaml、CLAUDE.mdの編集
- **記憶**: Mem0の内容閲覧・検索
- **成果物**: output/内の.mdファイルへのリンク・プレビュー
- **ログ**: 思考・実行ログの閲覧
- **コード修正**: 修正指示→サーバ→Claude Codeヘッドレス起動（`claude -p`）→結果を表示。ユーザーはClaude Codeの存在を意識しなくてよい

## 開発ルール

- **サーバはホットリロードで動かす** — ファイル変更時に自動再起動。手動再起動しない。ポート衝突を起こさない
- **テストは自動で回す** — コード修正後、Claude Codeが自分でテストを実行して結果を確認する。人間にブラウザ確認を頼まない
- **修正→テストのループを人間に挟ませない** — コード修正、テスト実行、結果確認、再修正までをClaude Codeが一貫して行う

## 開発手順

1. CLAUDE.mdのコンポーネント一覧と開発フェーズを確認する
2. 現在のフェーズに必要なコンポーネントの仕様書（`docs/spec_*.md`）を書く
3. コンポーネントを1つずつGitHubのIssueとして登録する。今必要ではない機能はIssue化しない
4. TDDで開発する（下記サイクル）
5. 1つの機能の実装とテストが完了すればプルリクエストを投げる
6. 人間が成果物を確認し修正依頼を返すかマージをする
7. コンポーネント一覧の状態を更新する

### TDDサイクル

1. **仕様書作成** — 何を作るか明確にする
2. **テストコード作成** — 仕様に基づいてテストを先に書く
3. **RED確認** — テストが失敗することを確認する（実装がないので当然失敗する）
4. **実装** — テストを通すための最小限のコードを書く
5. **GREEN確認** — 全テストが通ることを確認する
6. **リファクタリング** — テストが通る状態を維持しながらコードを整理する

## プロジェクト情報

### サーバー

- **ポート**: 8300（`run.py` の `DEFAULT_PORT`）
- **起動**: `start.bat`（ダブルクリック）。サーバーログ表示用のウィンドウが開き、ブラウザが自動で起動する
- **URL**: http://localhost:8300
- **ホットリロード**: uvicorn + watchfiles。`server/` 以下のファイル変更で自動再起動。サーバーの標準出力を `>nul` にリダイレクトするとリローダーが壊れるので禁止
- **停止**: サーバーウィンドウを閉じる

### GitHub

- **リポジトリURL**: https://github.com/masaru0ta/kobito_agent

## 注意事項
