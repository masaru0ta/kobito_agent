## 共通ルール

### エージェント間の相互作用

1. **独立した思考コンテキスト** — 各エージェントは完全に独立した思考を持つ。1つのLLM呼び出しで2役を演じさせない。
2. **相手の内面は見えない** — 見えるのは相手が発した言葉（チャット・トリガーメッセージ）だけ。
3. **記憶は共有しない** — 各エージェント専用のメモリ。知りたければトリガーで相手に質問する。
4. **役割分担は柔軟** — 基本的な役割はあるが、必要に応じて融通する。

### メモリ管理

- 各エージェントの記憶は `agent/{name}/memory/` に格納。
- Mem0による自動記憶機能（実装予定）。
- 記憶内容の詳細は各エージェント自身の判断に委ねる。

### 成果物管理

- すべての成果物は `.md` ファイルで `agent/{name}/output/` に出力。
- 各エージェントは `output/index.md` を必ず持つ。全ファイルへのリンク + 一行説明。
- ファイル新規作成時は `index.md` を更新。既存ファイルがあれば新規作成より更新を優先。

### エラー処理

- フォールバック処理の禁止。エラーは代替処理で誤魔化さず、明確に報告する。
- トリガー失敗時は、失敗の理由を明確に記録。

## トリガー設計

### ユーザーチャット
- Web UI からメッセージ送信 → エージェント起動
- 1対1 チャットまたは複数エージェント同時応答

### 定期トリガー（実装完了）
- cron 式スケジュール
- 例：10分ごとに自律思考サイクル実行
- アダムが10分間隔で自律思考実行中（2026-03-25 実装完了）

### エージェント間トリガー（未着手）
- 他エージェントからの明示的な呼び出し
- 例：Adam が Eden に「ユーザーニーズを聞いて」と依頼
- フェーズ5で実装予定

## 設計思想

### エージェントの本質
- エージェント = プロンプト（意思決定基準）+ ディレクトリ上のデータ（記憶・成果物）
- LLM は交換可能な思考エンジン
- UI は交換可能な接点
- **代替できないのは記憶だけ**

### ルール設計の原則
- 細かいルールで縛らない。むしろシンプルさを保つ。
- エージェント自身の判断に委ねる。
- 定義しない部分は、エージェント自身が判断・学習する。
